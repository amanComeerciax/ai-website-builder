# Phase 3: Build advanced context switching and workspace hydration (Lovable Core Workflow) - Research

## Objective
Research the architecture, dependencies, and integration points for building persistent Lovable-like workspaces. The primary challenge is transforming our current stateless UI (where the backend just streams out code without robust memory) into a 5-system database-driven architecture where every generation, log, and step is fully hydrated on page load.

## 1. Data Models (The Core Foundation)

The user provided exact schemas needed to power the pipeline:
1. **`Project`**: The workspace root. Contains the global `currentFileTree`, tying the active state together.
2. **`Message`**: Defines a single chat interaction (user prompt or AI response). Contains the `versionId` generated by that AI interaction, and `reasoning` chain-of-thought text.
3. **`Version`**: Immutable snapshot object representing a unique generation run. Contains the delta `diff`, the `fileTree` snapshot, and an `r2ZipKey`.
4. **`GenerationLog`**: Time-series log entry. The worker will emit these ("Installing framer-motion"). Frontend streams them live AND replays them for historical messages.

Mongoose will be the ORM. Foreign key refs (`ObjectId ref Project`) will manage relations.

## 2. Storage Architecture (Cloudflare R2)

Lovable stores the generated application payload in Cloudflare R2 as a `.zip` file for every single `Version`.
- **Dependency:** `@aws-sdk/client-s3` (compatible with R2)
- **Flow:** When BullMQ `aiWorker` finishes code generation, `Deploy Worker` (or `aiWorker` itself) will:
  1. `archiver` intercepts the in-memory files.
  2. Creates a zip buffer.
  3. Uploads to R2 using `projects/{projectId}/versions/{versionId}.zip`

## 3. Version Diffing (Timeline View)

Lovable shows exactly which files changed per version (green additions, red deletions) using GitHub-style diff viewing.
- **Dependency:** `jsdiff` or `diff` package in the worker/backend.
- **Comparison:** When an AI payload arrives, it compares the new `files` output with the last `Version`'s `fileTree`. Generates `Version.diff` = `[{ path, added, removed }]`.
- Frontend maps this `Version.diff` to the UI when a user looks at the version timeline.

## 4. Live Log Streaming & Replay

Currently, `server/routes/generate.js` streams BullMQ events directly to frontend SSE.
- **Modification Required:** Before or while streaming to the frontend via SSE, the worker MUST append logs to the `GenerationLog` MongoDB collection.
- Frontend's `/api/workspaces/:id` GET endpoint will return joined data: `Project` + `Messages` + their associated `GenerationLogs`.
- If the frontend realizes a message is historical (`message.status === 'done'`), it plays the array of `GenerationLogs` instead of listening to SSE.

## 5. Deployment Previews (Netlify IFrames)

Every version natively targets Netlify.
- **Dependency:** Netlify REST API (via `fetch` or `netlify` package).
- Post-generation, the zip artifact is POSTed directly to `/api/v1/sites` for Netlify deployments.
- Output URL stored in `Version.previewUrl` and surfaced to an iframe in `PreviewPanel`.

## Architectural Integration Points
- **BullMQ Payload Injection:** The job payload in `routes/generate.js` must pass `chatHistory` (fetched from Message DB) so the AI retains historical project memory.
- **REST Endpoints Requirements:** 
  - `POST /api/workspaces/:id/messages` - Creates user messages, kicks off AI BullMQ worker.
  - `GET /api/workspaces/:id` - Fetches the full context (hydrates frontend).
  - `GET /api/projects` - To list User projects.

## Validation Architecture (Dimension 8)
- [x] **Data Integrity:** Fetching a `/workspace/:id` successfully returns its correct history.
- [x] **R2 Artifact:** Generation succeeds and an S3 PutObject verifies against bucket.
- [x] **State Resumption:** User can close tab, reopen `/workspace/:id` and IDE visually loads the previous chat + file tree exactly as left off.
